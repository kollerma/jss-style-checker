# Quickstart: Running `jss-lint init`

**Audience**: a JSS author starting from a manuscript with no
config; a contributor extending the init logic.

## For a JSS author

### Step 0: bootstrap a config

```sh
cd path/to/my-paper
jss-lint init manuscript.tex
```

Output:

```
Found 87 violations across 14 rules.
Focus first on these 5 high-precision must-fix rules (23 violations):

  JSS-MARKUP-001  (8 violations, 8 lines, precision 0.96)
  JSS-CITE-002    (6 violations, 5 lines, precision 0.94)
  JSS-NAME-001    (5 violations, 4 lines, precision 0.93)
  JSS-CAP-002     (3 violations, 3 lines, precision 0.92)
  JSS-WIDTH-001   (1 violation,  1 line,  precision 1.00)

Suppressed 9 noisy rules (precision < 0.90); see .jss-lint.toml.

Wrote .jss-lint.toml
```

The generated `.jss-lint.toml` looks like:

```toml
journal = "jss"

# auto-generated by `jss-lint init` on 2026-05-03
# threshold = 0.90 (matches Constitution §VI)

ignore_rules = [
  # JSS-MARKUP-005: precision 0.74 (corpus-wide), below 0.90
  "JSS-MARKUP-005",
  # JSS-NAME-007: precision 0.82, below 0.90
  "JSS-NAME-007",
  # ... 7 more rules
]
```

### Step 1: lint as normal

```sh
jss-lint manuscript.tex
```

You'll see the 23 must-fix violations only; the noisy rules
have been suppressed by the generated config.

### Step 2: review the suppressions

Open `.jss-lint.toml` in your editor. Each suppressed rule
has a comment explaining why. Remove any rule from
`ignore_rules` if you want to see its violations again.

### Re-running `init`

```sh
jss-lint init manuscript.tex
# error: refusing to overwrite .jss-lint.toml; pass --force to overwrite

jss-lint init --force manuscript.tex
# overwrites; the previous config is gone
```

### Audit mode

```sh
jss-lint init --dry-run manuscript.tex
```

Prints the proposed config to stdout and the summary to
stderr. No file is written.

### Custom threshold

```sh
jss-lint init --threshold 0.95 manuscript.tex
```

Tighter — only rules with corpus precision ≥ 0.95 are
classified `must-fix`.

## For a contributor

### Where things live

```text
src/texlint/init.py                            # the module
src/texlint/cli.py                             # subcommand wiring
tests/unit/test_init.py                        # classification + TOML
tests/integration/test_init_cli.py             # end-to-end CLI
tests/fixtures/init/                           # manuscript + DB fixtures
```

### Run the tests

```sh
pytest tests/unit/test_init.py tests/integration/test_init_cli.py -v
```

### Adding a new field to the generated config

1. Decide whether it's per-rule (goes inside `ignore_rules`) or
   global (top-level key).
2. Update `init._render_toml` to emit the new field.
3. Regenerate the goldens for the noisy / clean fixtures.
4. Update `data-model.md §2` with the new shape.

### Common pitfalls

- **The DB lookup is slow**: each rule is one SQLite query; for
  58 rules, this is sub-millisecond. If you see slowness, the
  cause is the lint pass, not the DB.
- **Comment / array-element drift**: the renderer emits the
  comment + element pair atomically. If you see an array
  element without its preceding comment, you've added a code
  path that constructs the array imperatively rather than via
  `_render_toml`. Don't.
- **Atomic write left a temp file**: the `finally` block in
  `_atomic_write` should clean up. If a temp file remains,
  there is a bug; the test suite should reproduce.
