# Feature Specification: JSS Rule Catalogue — From 5 Smoke Rules to ~50 Production Rules

**Feature Branch**: `003-jss-rule-catalogue`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "Expand the JSS journal module to cover the complete JSS style guide, growing from the ~5 smoke-test rules in Step 1 to approximately 50 rules across roughly 16 categories. This spec is primarily an enumeration exercise: most of the value is in producing an authoritative, cross-referenced catalogue derived from four sources — the style guide, the author manuscript-preparation section, `article.tex`, and `jss.cls`."

## Clarifications

### Session 2026-04-23

- Q: How is the rule catalogue stored — markdown table, machine-readable data file, or both? → A: **Data file as source of truth, markdown rendered from it.** The catalogue is authored as a structured data file (YAML or TOML, to be pinned in `/speckit.plan`) at `specs/003-jss-rule-catalogue/catalogue.yaml`; a small rendering step produces `catalogue.md` for human review. Tests consume the data file directly and enforce the structural invariants in FR-001 through FR-006 (unique `rule_id`s within and across categories, `authority` values inside the four-value enum, every `article_tex` / `jss_cls` `authority_ref` resolving against the vendored template, every `severity` inside `{error, warning, info}`, every `inspects` value inside the declared `ParsedDocument`-field enum, every `auto_fixable` a boolean). The markdown file is a rendered artefact, not a second source of truth; a CI check refuses commits where the rendered markdown drifts from the data file.
- Q: How do the Step 1 smoke rules (`bib_001_year.py`, `cite_001_emph.py`, `src_001_width.py`) move into the new per-category module layout? → A: **Delete old, rewrite in place.** The first PR that introduces a category containing a pre-existing smoke rule deletes the old one-rule-per-file module in the same change that adds the category module and moves the associated test into `tests/unit/rules/test_<category>.py`. No compatibility shims, no parallel registration paths. Git history preserves the old file via `git log --follow`; the repository carries one authoritative module per category from the moment the category ships.
- Q: What is the minimum number of labelled violations a rule needs before its corpus precision counts against the ≥90% gate? → A: **N=10 minimum, and the eval-jss corpus grows to approximately 50 papers** so that floor is attainable. A rule with fewer than 10 labelled violations is reported by `eval-jss report` as "not yet measured" and is not gated by the 90% threshold (same bucket spec 002 §SC-003 already defines). To make the N=10 floor enforceable for most rules, the eval-jss corpus target — which spec 002 explicitly scopes as "an operational knob under this spec, not a versioned deliverable" — grows from the Step 2 starter of ~10 CRAN vignettes to approximately 50 papers by the time the rule catalogue stabilises. This scaling is in-scope for spec 003's task list (corpus grows as categories land) and stays within spec 002's flexibility envelope; spec 002 itself is not amended by this clarification.
- Q: What deviation from the "~50 rules / ~16 categories" targets is acceptable without a spec amendment? → A: **No fixed tolerance — the targets are aspirational, not contractual.** The final rule count and category count are whatever the authorities (`jss.cls`, `article.tex`, style guide, author instructions) warrant, filtered through the authority-citation requirement (FR-002) and the precision gate (FR-015). There is no upper or lower bound that triggers review on its own; review is triggered by *what* is in or out of the catalogue, not *how many* entries it has. A catalogue that lands at 38 rules because 12 proposals failed the authority or precision gate is a success, not an under-delivery; a catalogue that lands at 65 because the authorities justify them is equally valid. The "~50 / ~16" numbers remain useful as planning anchors — if the count drifts by more than a factor of two from those anchors, that is a signal worth a clarify pass, but it is not a hard contract.
- Q: Which rule categories are required to consume the shared term list at `src/texlint/journals/jss/terms.py` — the fixed three (`capitalization`, `naming`, `house_style`) or any category that legitimately needs token canonicalisation? → A: **Open list — any rule module that checks the canonical casing, canonical form, or proper-noun spelling of a language, package, or code-like token already in `terms.py` MUST consume it.** `capitalization`, `naming`, and `house_style` remain the definite consumers; `references`, `markup`, `citations`, `abbreviations`, and any future category that inspects a term the list already knows about MUST route through `terms.py` rather than re-declaring the string. No per-category exceptions; no parallel term lists anywhere in `src/texlint/journals/jss/`. A consistency test MUST fail the build when a rule module hard-codes a token whose canonical form is already recorded in the shared list.
- Q: Should rule-module implementation (per-category rollout + precision gate) land in this spec or in a successor? → A: **Scope narrowed to catalogue-only; rule-module rollout deferred to spec 004.** This spec's deliverable is the authority-cited catalogue at `catalogue.yaml`, the shared term list at `terms.py`, and the tooling that validates + renders them. Implementation of `src/texlint/journals/jss/rules/<category>.py` modules, their unit tests, their fixtures, and the per-category corpus precision gate are deferred to a successor spec (tentatively "spec 004 — JSS Rule Modules"). The reviewer's substantive feedback on 2026-04-23 (drop `JSS-CITE-001`, refine heuristics on seven rules, add nine new rules, defer `JSS-ABBR-002`, split `JSS-REFS-002` into tight + loose) is absorbed into `catalogue.yaml` before freezing spec 003; spec 004 implements against the stable catalogue rather than chasing a moving target. Phase 5 work (the `citations` rule module committed 2026-04-23) has been **reverted** (`git revert 3379299`); the Step 1 smoke rules (`bib_001_year.py`, `cite_001_emph.py`, `src_001_width.py`) remain in place as they did before Phase 5, and will be retrofitted by spec 004's category rollout. FR-015 through FR-020, SC-003, SC-004, and User Story 2 are all re-scoped to "deferred to spec 004" by this clarification.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A single authoritative rule catalogue traceable to source (Priority: P1)

A linter maintainer, a reviewer, or a JSS author opening the repository wants to know, without reading Python code, **which JSS style rules the tool enforces, why they are rules, and where in the official JSS materials each one is justified**. They open `specs/003-jss-rule-catalogue/catalogue.md` and see one table row per rule with a stable identifier, category, severity, one-line description, the authoritative source (`jss.cls`, `article.tex`, style guide prose, or author instructions), a section/line anchor within that source, an example violation, and an example fix. Every row cites at least one authority; rows without a citation are marked for discussion during `/speckit.clarify` rather than silently shipping.

**Why this priority**: The enumeration is itself the primary deliverable of Step 3 — the user said so explicitly. Without a cross-referenced catalogue, the rest of the work (module layout, unit tests, corpus tuning) is just pattern-matching against an unwritten brief. Once the catalogue exists, every subsequent task (implement the `citations` module, write tests for `JSS-CITE-003`) has a single source of truth. Authority tracing is what distinguishes this catalogue from a paraphrase of the prose guide: when two authorities disagree, the rule is pinned to the higher-authority source (`jss.cls` > `article.tex` > style guide prose > author instructions).

**Independent Test**: A reviewer can take any rule row from the catalogue and, by following the `authority_ref` alone (a section anchor or a line number in the vendored copy of `article.tex`/`jss.cls`), locate the passage in the source that justifies the rule without needing to read Python code or ask the author. A rule whose justification cannot be reproduced from its authority_ref is a spec bug, not an implementation bug.

**Acceptance Scenarios**:

1. **Given** the catalogue is complete, **When** a reader picks any rule at random, **Then** the row carries a non-empty `authority` value drawn from the four-value enum, a resolvable `authority_ref`, a one-line `description`, a synthetic `example_violation` LaTeX fragment, and an `example_fix` LaTeX fragment that renders the violation resolved.
2. **Given** the catalogue includes a rule whose only justification is inferred convention (no source anchor), **When** the catalogue is reviewed, **Then** that rule is explicitly marked for `/speckit.clarify` resolution and does not count toward the "ready to implement" subset.
3. **Given** two authorities disagree (e.g., the prose guide says "use `\proglang{}` for language names" but `jss.cls` also defines `\code{}` and a reader might plausibly use either), **When** a rule is pinned, **Then** the authority chain (`jss.cls` > `article.tex` > style guide > author instructions) resolves the conflict and the losing source is noted in the rule's row so the disagreement is not lost.
4. **Given** a JSS author visits the repository to understand why the linter complained, **When** they look up the rule id from their terminal output in the catalogue, **Then** they can read an example violation they recognise and an example fix they can copy, without needing the source code.

### User Story 2 — Category-by-category rollout gated on corpus precision (Priority: P1) *(deferred to spec 004)*

**Status (2026-04-23)**: this user story is **deferred to spec 004 — JSS Rule Modules**. See the Session 2026-04-23 clarification that narrows spec 003 to catalogue-only scope. The narrative and acceptance scenarios below describe the eventual rollout but are not in-scope for spec 003.

---

A linter developer implementing the catalogue does not ship all 50 rules in one batch. They work category by category, in a deliberate order (`citations` and `references` first, where the style guide is most explicit; `typography` and `capitalization` last, where false positives bite hardest). For each category, they implement the rules with unit tests, run `eval-jss scan` over the corpus, label the new violations, run `eval-jss report`, and only consider the category "done" when every rule in it meets the ≥90% precision bar set by Constitution Principle VI. Rules below threshold are refined or relaxed before the category commits.

**Why this priority**: A 50-rule catalogue committed in one shot would be impossible to review and would land a wall of false positives on any real manuscript. The category-by-category loop — matching the infrastructure built in Step 2 — is the quality gate that keeps the linter's reputation intact. It is also the loop the Constitution explicitly endorses for rule refinement, so shipping rules without it would violate Principle VI before the catalogue even closes.

**Independent Test**: For any single category (say, `citations`), a linter developer starting from `main` can add that category's rules, run unit tests to green, run `eval-jss scan && eval-jss human-review && eval-jss report`, and observe that every rule in the category exceeds 90% precision on the corpus before they open a commit. A rule that ships at 87% must either be tightened, relaxed, or removed — not hidden behind aggregated numbers.

**Acceptance Scenarios**:

1. **Given** the `citations` category has been chosen as the first to implement, **When** the developer lands its rules, **Then** each rule has a unit test file with 100% branch coverage under `tests/unit/rules/test_citations.py`, and every rule's precision on the corpus (per `eval-jss report`) is ≥90% before the commit.
2. **Given** a rule in a shipped category drops below 90% after a corpus addition, **When** the next `eval-jss report` runs, **Then** that rule is visibly flagged under threshold and the category loses its "done" status until the rule is refined, relaxed, or retired.
3. **Given** the `typography` and `capitalization` categories are intentionally scheduled last, **When** they are implemented, **Then** they benefit from a larger corpus and their threshold tuning is informed by real violation distributions rather than synthetic fixtures alone.
4. **Given** a category comprises rules that reference the shared term list (e.g., `capitalization` cross-references `terms.py` for proper capitalisation of `R`, `Python`, `SAS`, `randomForest`), **When** a term is added or corrected, **Then** rules pick up the change through a single source of truth rather than duplicating the string across modules.

### User Story 3 — Offline, reproducible rule derivation from vendored official sources (Priority: P2)

A contributor with no internet access, or one validating a rule six months after the official JSS template was last updated, wants to derive rules from exactly the sources the catalogue was built against. They find `docs/jss-template/` checked into the repository, containing the current `article.tex` and `jss.cls` from `jss-article-tex.zip` plus a provenance note (date fetched, upstream URL, SHA256). Every `authority_ref` in the catalogue points into these vendored files by line number, so rule derivation is reproducible offline.

**Why this priority**: Without vendoring, the catalogue silently rots: upstream JSS updates `jss.cls`, the line numbers in `authority_ref` drift, and nobody notices until a contributor opens a stale reference. With vendoring plus an annual re-fetch-and-diff, upstream changes are caught on purpose rather than by accident, and the rules-to-sources chain stays tight. Vendoring also removes a class of CI flakes (upstream server hiccups during rule work) without turning the harness into a web scraper.

**Independent Test**: A contributor in an offline environment can clone the repository, open `docs/jss-template/article.tex`, and follow any catalogue `authority_ref` (line number or section anchor) to the justifying passage. An annual diff job comparing the vendored copy against a fresh upstream download surfaces any changed lines and prompts a catalogue review; with no network, no diff runs, but the catalogue remains usable against the last known good template.

**Acceptance Scenarios**:

1. **Given** the repository has been cloned to a machine with no internet access, **When** a contributor opens `docs/jss-template/`, **Then** they find the full unpacked contents of `jss-article-tex.zip` plus a provenance README recording the upstream URL, the fetch date, and a SHA256 of the zip.
2. **Given** an `authority_ref` in the catalogue of the form `article.tex:142` or `jss.cls:§markup:\pkg`, **When** a reader resolves it, **Then** the referenced line or section is present in the vendored file and plausibly justifies the rule.
3. **Given** upstream has published a new `jss.cls` since the last vendor, **When** the annual re-fetch runs, **Then** a diff is produced so a maintainer can decide per-rule which catalogue rows (if any) need an updated anchor or a changed rule body.
4. **Given** a rule cites the style guide web page (`https://www.jstatsoft.org/style`) or the author instructions page, **When** the `authority_ref` is recorded, **Then** it captures a section/anchor stable enough to survive minor page edits (e.g., a heading name) rather than a line count that the web page does not expose.

### User Story 4 — Shared term list powering cross-rule knowledge (Priority: P2)

Several rule categories (`capitalization`, `naming`, `house_style`) need to know the same facts: `R` is a language, `Python` is capitalised with uppercase P, `SAS` is always all-caps, `randomForest` is the canonical spelling of the CRAN package. A rule developer adding `caret` to the catalogue should not have to edit three modules. They add the term once to `src/texlint/journals/jss/terms.py`, and every rule that cross-references the list picks it up automatically.

**Why this priority**: Duplicated strings across rule modules are how rules silently drift out of sync. A shared term list is a small piece of infrastructure (a few curated dicts) that pays for itself the first time a term needs correcting. It is not a blocker for the first category to ship — `citations` does not need it — which is why it is P2 and can land in parallel with the early categories rather than before them.

**Independent Test**: With the term list populated with two programming languages and two R packages, a unit test for `capitalization` and a unit test for `naming` both consume the same entries and both fail when a stale duplicate is introduced elsewhere. Removing the shared list and reintroducing duplicates causes the cross-module consistency test to fail.

**Acceptance Scenarios**:

1. **Given** the term list contains `{"R": language, "Python": language, "randomForest": r_package}`, **When** `capitalization` rules check a manuscript that writes `r` in prose, **Then** they flag it; **When** they check `R` itself, they do not.
2. **Given** a contributor adds a new package to the list, **When** the next `eval-jss scan` runs, **Then** every rule that cross-references the list gains coverage for that term without any rule module being edited.
3. **Given** a term's status is ambiguous in the real world (e.g., `jags` vs `JAGS`), **When** the list is queried, **Then** the canonical form is returned alongside an optional "also accepted" set so rules can distinguish outright errors from style-guide preferences.

### Edge Cases

- **A rule is obvious from the prose guide but has no anchor in `jss.cls` or `article.tex`**: record it with `authority = style_guide` and the deepest stable anchor the page offers (section id, heading text). It still counts as cited. Only rules with *no* source anchor anywhere are flagged for `/speckit.clarify`.
- **Two rules in different categories collapse to the same check**: pick the category whose authority is higher; the duplicate row is removed and the chosen rule's description notes the other category it could have lived in. The catalogue is a rule catalogue, not a concept catalogue — a check fires from exactly one place.
- **A rule is plausible but the authority is silent**: the rule is *not* added to the catalogue. Inferred-convention rules are a separate conversation in `/speckit.clarify` and must earn an authority before implementation. No speculative rules ship.
- **Upstream `jss.cls` renames a macro between fetches**: the annual diff surfaces the rename; affected catalogue rows get updated anchors and, if behaviour changed, updated rule bodies. A rename that breaks an existing rule is a blocker, not a silent drift — the catalogue must remain internally consistent with the vendored copy.
- **A proposed rule would push an entire category below the 90% precision floor even at 100% individual precision** (e.g., because the corpus legitimately violates it constantly): the rule is relaxed (stricter fixtures, narrower match) or retired before the category commits. The threshold is non-negotiable; the rule content is not.
- **A category covers an area the corpus barely exercises** (e.g., `abbreviations` in mathematical preprints): the category is still implemented, but "not exercised" rules are accepted for that commit on the explicit understanding that precision is monitored as the corpus grows. This is surfaced by the `eval-jss report` "not exercised" bucket (see spec 002 §SC-003).
- **The style guide web page changes an anchor name** (e.g., a heading rename): catalogue rows citing that anchor become unresolvable. Annual re-fetch is the recovery path; between fetches, the vendored copies of `article.tex` and `jss.cls` carry the bulk of the anchor burden precisely because they are stable.
- **A rule inspects BibTeX but the behaviour is mechanical (missing key) vs content-level (missing DOI)**: mechanical rules live in `bibtex`, content rules live in `references`. The boundary is pinned so `JSS-BIBTEX-001` is not interchangeable with `JSS-REFS-001` — see the Category Boundaries pin in Assumptions.

## Requirements *(mandatory)*

### Functional Requirements

**Catalogue structure and traceability**

- **FR-001**: The feature MUST produce an authoritative rule catalogue whose **source of truth** is a machine-readable data file at `specs/003-jss-rule-catalogue/catalogue.yaml` (YAML or TOML; the concrete format is pinned in `/speckit.plan`). Each entry MUST carry exactly these fields: `rule_id`, `category`, `severity`, `description`, `authority`, `authority_ref`, `example_violation`, `example_fix`, `inspects` (which `ParsedDocument` field the rule reads), `auto_fixable` (boolean flag only — implementation deferred to Step 5), and an optional `notes` field. A companion human-readable artefact `specs/003-jss-rule-catalogue/catalogue.md` MUST be **generated** from the data file by a deterministic rendering step; it is a rendered artefact and MUST NOT be hand-edited. A CI check MUST refuse commits where the rendered markdown drifts from the data file.
- **FR-002**: Every catalogue row MUST cite at least one `authority` drawn from the four-value enum `{style_guide, author_instructions, article_tex, jss_cls}`. A row whose authority is empty is a spec bug; a row whose justification is inferred convention MUST be flagged for `/speckit.clarify` discussion and MUST NOT be marked ready for implementation.
- **FR-003**: When authorities disagree, the catalogue MUST pin the rule to the highest-authority source using the order `jss_cls > article_tex > style_guide > author_instructions` and MUST record the losing source in a free-text `notes` field on the row so the disagreement is not lost between spec revisions.
- **FR-004**: Rule identifiers MUST follow the scheme `JSS-<CATEGORY>-NNN` with a three-digit zero-padded counter (e.g., `JSS-CITE-001`, `JSS-PREAMBLE-012`). Counters are per-category and MUST be stable across the catalogue's lifetime — a retired rule's id is never reused.
- **FR-005**: The category set MUST cover the distinct concerns the JSS authorities address. A **planning anchor** of approximately 16 categories is used to frame the work; the current proposed set is `preamble`, `structure`, `markup`, `citations`, `references`, `bibtex`, `naming`, `capitalization`, `typography`, `abbreviations`, `code_style`, `code_width`, `operators`, `crossrefs`, `house_style`. The actual final category set is determined by what the authorities warrant — categories MAY be added, merged, or dropped as enumeration reveals real coverage boundaries, and no fixed upper or lower bound on category count is imposed. The category list MUST appear in a pinned location (this spec, its clarifications, or the catalogue data file itself) before any category begins implementation.
- **FR-006**: Catalogue scale is **authority-driven, not count-gated.** The "~50 rules" figure is a planning anchor, not a contractual target. A rule is included iff it cites at least one authority (FR-002) and clears the precision gate (FR-015); a rule is excluded iff it fails either gate. There is no upper or lower bound on rule count that triggers review on its own. The numerical anchor is retained in this spec only so readers have an order-of-magnitude expectation; if the final count drifts more than a factor of two from ~50 (i.e., below ~25 or above ~100) that is itself an observation worth a clarify pass, but the catalogue's validity is not defined by hitting any count.

**Source vendoring and reproducibility**

- **FR-007**: The official JSS template distribution MUST be vendored into the repository under `docs/jss-template/` (or an equivalent checked-in path noted in the catalogue). The vendored copy MUST be the unpacked contents of `jss-article-tex.zip`, accompanied by a provenance record containing the upstream URL, the fetch date, and a SHA256 of the downloaded zip.
- **FR-008**: Every catalogue row whose `authority` is `article_tex` or `jss_cls` MUST have an `authority_ref` that resolves against the vendored copy (line number, section anchor, or macro name). A reader MUST be able to navigate from the `authority_ref` to the justifying passage without network access.
- **FR-009**: The repository MUST document an annual re-fetch cadence: diff the current vendored copy against a fresh upstream download, record the diff in a changelog note, and update affected `authority_ref` values and rule bodies in the same commit.
- **FR-010**: Rules whose `authority` is `style_guide` or `author_instructions` MUST record an `authority_ref` that points to a stable section/anchor on the relevant web page (e.g., heading id) rather than a line number the page does not expose.

**Rule metadata**

- **FR-011**: Every rule MUST declare which `ParsedDocument` field it inspects (one or more of the AST under `tex_files`, the parsed BibTeX under `bib_files`, or raw source lines). The declaration lets the plan stage budget parser work and lets reviewers anticipate false-positive surface area.
- **FR-012**: Every rule MUST declare an expected severity drawn from the set `{error, warning, info}`. The default mapping — objectively-wrong rules default to `error`, stylistic rules default to `warning`, rules that are purely informational default to `info` — is pinned in the Assumptions section and is the starting point for every row; deviations are recorded in the rule's `notes`.
- **FR-013**: Every rule MUST declare an `auto_fixable` boolean. The boolean is a flag only — fix payloads (`FixSuggestion`) are explicitly Step 5 and MUST NOT ship in this spec. An `auto_fixable = true` flag is a promise to Step 5 that the fix is mechanically expressible; a `false` flag declares the rule as diagnosis-only.
- **FR-014**: The rule `formats` field (the filter that scopes a rule to `.tex`, `.bib`, or both) MUST be left at its default `None` (all formats) in this spec. Format-specific narrowing is Step 4 scope.

**Per-category quality gate** *(FR-015 through FR-020 are deferred to spec 004; see Session 2026-04-23 clarification)*

- **FR-015** *(deferred to spec 004)*: Each category's implementation MUST be gated on corpus precision measured via `eval-jss report`: every rule in the category with at least **10 labelled violations** on the corpus MUST reach ≥90% precision before the category is considered done. Rules with fewer than 10 labelled violations are reported as "not yet measured" (the bucket spec 002 §SC-003 already defines) and MUST NOT be used as evidence of either passing or failing the gate — they ship flagged as undercounted and are re-evaluated as the corpus grows. Rules above the N=10 floor and below 90% precision MUST be refined or relaxed; if they cannot clear the bar, they MUST be retired before the category commits. This is the direct application of Constitution Principle VI and is non-negotiable.
- **FR-016** *(deferred to spec 004)*: Each category MUST ship with a unit test file at `tests/unit/rules/test_<category>.py` exercising 100% branch coverage of its rule module. Unit tests MUST use small synthetic LaTeX strings and dedicated fixture files under `tests/fixtures/violations/`; they MUST NOT depend on the corpus (the corpus is the downstream precision gate, not the upstream unit-test target).
- **FR-017** *(deferred to spec 004)*: Categories MUST ship in an order that front-loads the highest-signal, best-specified ones and defers the most false-positive-prone ones: `citations` and `references` first; `typography` and `capitalization` last. The full ordering is fixed during `/speckit.clarify` and recorded in `tasks.md`, not here.

**Module layout** *(FR-018 and FR-019 are deferred to spec 004; FR-020 remains in-scope because `terms.py` shipped with the catalogue)*

- **FR-018** *(deferred to spec 004)*: Rules MUST live one Python module per category under `src/texlint/journals/jss/rules/`, each exporting `check_*` callables conforming to `Callable[[ParsedDocument], Iterator[Violation]]` (the existing linter contract). Rules from different categories MUST NOT live in the same module.
- **FR-019** *(deferred to spec 004)*: The `JSSJournal` journal plugin MUST assemble the active rule set from a single well-known entry point (`JSSJournal.categories()` or equivalent) — rules MUST NOT be registered ad-hoc from module-level imports. This keeps "which rules does the linter run?" answerable by reading one function.
- **FR-020**: A shared term list MUST live at `src/texlint/journals/jss/terms.py` covering programming languages, statistical packages, and code-like tokens that need canonical casing or proper-noun form. **Any** rule module that inspects the canonical casing, canonical form, or proper-noun spelling of a token already recorded in the list MUST consume the list; it MUST NOT re-declare the term locally. This includes (but is not limited to) the `capitalization`, `naming`, `house_style`, `references`, `markup`, `citations`, and `abbreviations` categories whenever they touch a token the list knows about. Parallel or shadow term lists anywhere under `src/texlint/journals/jss/` are forbidden. A consistency test MUST fail the build when a rule module hard-codes a token whose canonical form is already recorded in the shared list.

**Out of scope**

- **FR-021**: `.Rnw` and `.Rmd` MUST NOT be supported in this spec. Rules target `.tex` and `.bib` only. Format narrowing for these formats (Step 4) MUST NOT leak into this spec.
- **FR-022**: Fix payloads (`FixSuggestion`) MUST NOT ship in this spec. Every rule records `auto_fixable` as a flag; the flag's value guides Step 5 prioritisation but produces no executable fix here.
- **FR-023**: Package-ecosystem validation (e.g., hitting CRAN to verify that `randomForest` is a real package) MUST NOT ship in this spec. The static term list (FR-020) is the sole source of package-name truth for now; CRAN-backed validation is a documented follow-up.

**Retrofit of Step 1 smoke rules**

- **FR-024**: The three Step 1 smoke-test modules (`src/texlint/journals/jss/rules/bib_001_year.py`, `cite_001_emph.py`, `src_001_width.py`) and their tests MUST be migrated to the per-category layout using a **delete-and-rewrite** strategy: the PR that first introduces a category subsuming a smoke rule (e.g., `citations` subsumes `cite_001_emph.py`) MUST delete the old module and its test in the same change that adds `rules/<category>.py` and `tests/unit/rules/test_<category>.py`. Compatibility shims, re-exports, or parallel registration paths MUST NOT be introduced. Git history preserves the provenance via `git log --follow`; the repository carries one authoritative module per category from the moment the category ships.

### Key Entities

- **Rule Catalogue**: The durable, source-cited enumeration of every check the JSS journal plugin ships. One row per rule, with the schema pinned in FR-001. The catalogue is the contract against which rule modules and unit tests are written; if the catalogue says a rule exists, a module check and a test MUST follow (or the catalogue row MUST be removed).
- **Rule Row**: A single entry in the catalogue — `rule_id`, `category`, `severity`, `description`, `authority`, `authority_ref`, `example_violation`, `example_fix`, `inspects`, `auto_fixable`, optional `notes`. The row is the unit of review, the unit of traceability, and the unit of quality-gate accounting.
- **Authority**: One of `{style_guide, author_instructions, article_tex, jss_cls}`, ordered `jss_cls > article_tex > style_guide > author_instructions`. The authority establishes why a rule exists; the `authority_ref` establishes where.
- **Category**: A bucketing of rules by concern (`preamble`, `citations`, ...). A category is a commit boundary (an implementation lands as a whole), a precision-gate boundary (every rule in the category must clear the threshold), a module boundary (one `.py` file), and a test boundary (one `test_*.py` file).
- **Vendored JSS Template**: The unpacked contents of `jss-article-tex.zip` checked into `docs/jss-template/`, with provenance (upstream URL, fetch date, SHA256). The vendored copy is the target of every `article_tex` and `jss_cls` `authority_ref`.
- **Term List**: The shared dictionary at `src/texlint/journals/jss/terms.py` of programming languages, statistical packages, and code-like tokens with their canonical capitalisation. Consumed by `capitalization`, `naming`, and `house_style` rules.
- **Precision Gate**: The Step 2 `eval-jss report` number each rule must meet (≥90%) before its category commits. The gate lives in spec 002; this spec uses it as an external contract.
- **ParsedDocument** *(referenced, defined in spec 001)*: The per-paper AST/BibTeX/source-lines bundle passed to each rule check. Every catalogue row declares which of its fields the rule inspects (FR-011).
- **Violation** *(referenced, defined in spec 001)*: The existing per-check output type; no changes to the type in this spec.
- **FixSuggestion** *(referenced, Step 5 scope)*: The fix payload type. Mentioned only to be explicitly excluded (FR-022); `auto_fixable` is a flag for Step 5, not an implementation here.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The catalogue enumerates the rules the JSS authorities warrant — the planning anchor is ~50 rules across ~16 categories, but the actual final count is driven by what the authorities justify and what the precision gate admits. Every entry in the data-file source of truth has a non-empty `authority`, a resolvable `authority_ref`, a one-line `description`, a synthetic `example_violation`, and an `example_fix`. Zero catalogue entries lack an authority at the moment a category begins implementation. A CI test parses the data file and fails the build when any structural invariant (unique `rule_id`s, enum membership for `authority` / `severity` / `inspects`, resolvability of `authority_ref` against `docs/jss-template/`, rendered-markdown-matches-data-file) is violated.
- **SC-002**: For every rule whose `authority` is `article_tex` or `jss_cls`, a reader can navigate from the catalogue row's `authority_ref` to the exact passage in `docs/jss-template/` without network access and verify that the referenced passage justifies the rule.
- **SC-003** *(deferred to spec 004)*: Every shipped category meets the per-rule precision floor: for every rule with ≥10 labelled violations in a category marked done, `eval-jss report` shows precision ≥90% on the corpus at the commit that closes the category. Rules with fewer than 10 labelled violations ship as "not yet measured" rather than passing or failing the gate. Rules above the N=10 floor and below 90% do not ship; they are refined, relaxed, or retired first.
- **SC-004** *(deferred to spec 004)*: Every shipped category has a unit test file at `tests/unit/rules/test_<category>.py` with 100% branch coverage of its rule module, measured by the project's coverage tool. A category without full branch coverage is not done.
- **SC-005**: A contributor asking "what JSS rule is this?" can answer it from the terminal output alone by looking up the `rule_id` in the catalogue. They do not need to read Python source, ask a maintainer, or consult the JSS website.
- **SC-006**: Terms in the shared list are single-sourced. A test or lint check ensures that no rule module anywhere under `src/texlint/journals/jss/rules/` re-declares a term that already lives in `terms.py` — not just `capitalization`, `naming`, and `house_style`, but every category that touches a term the list knows about (`references`, `markup`, `citations`, `abbreviations`, and any future category that cross-references a known token). Parallel or shadow term lists are forbidden; the build fails when one is introduced.
- **SC-007**: When `jss.cls` changes upstream (detected by the annual re-fetch and diff), affected catalogue rows are identified in the same change set that updates the vendored copy. No catalogue row carries an `authority_ref` that does not resolve in the currently-vendored template.
- **SC-008**: The category order pinned during `/speckit.clarify` schedules `citations` and `references` first and `typography` and `capitalization` last, matching the signal-to-noise profile on the corpus. A different order requires a documented justification in the clarifications log.

## Assumptions

- **Sources of truth, ordered**: The rules are derived from four JSS sources in strict priority order: `jss.cls` > `article.tex` > style guide prose (`https://www.jstatsoft.org/style`) > author instructions (`https://www.jstatsoft.org/authors#manuscript-preparation`). The class file is highest because it defines the commands that the journal's typesetting pipeline actually runs; the prose guide describes the desired outcome but is silent on the mechanics. Rules MUST NOT hallucinate commands — if `jss.cls` does not define it, no rule can require it.
- **Vendoring location**: Default vendor path is `docs/jss-template/`. Actual path is confirmed during `/speckit.plan`; any change is noted in the catalogue's header.
- **Default severity mapping**: `error` is the default for objectively-wrong conditions (missing preamble macros, malformed citations, undefined cross-references); `warning` is the default for stylistic conditions (capitalisation, code width, typographic dashes); `info` is the default for informational findings (rule did fire, no action required). Each rule row carries its own severity; the default is the starting point, not a ceiling, and deviations are recorded in the row's `notes`.
- **Category boundary — references vs. bibtex**: `references` holds content-level rules about BibTeX *entries* (missing DOI on a journal article, incomplete author list, wrong casing on a proper noun); `bibtex` holds mechanical rules about BibTeX *syntax* (missing key, malformed field, duplicated key, entries BibTeX would reject). A content-level rule never lives in `bibtex`; a mechanical rule never lives in `references`. This pin is the answer to the user's `/speckit.clarify` candidate and is treated as fixed unless `/speckit.clarify` surfaces a concrete counterexample.
- **Category order**: `citations` and `references` ship first — the style-guide language around citation formatting is the most explicit and the signal-to-noise on the corpus is the highest. `typography` and `capitalization` ship last — they are the most false-positive-prone, and tuning their thresholds benefits from a larger labelled corpus. Middle categories are ordered during `/speckit.clarify` by a mix of authority density and implementation effort.
- **Package ecosystem knowledge**: No CRAN lookup in this spec. The shared term list (`terms.py`) is the sole source of package-name truth. Dynamically validating against CRAN (e.g., asking `https://crandb.r-pkg.org/` whether `randomForest` is a package) is explicitly out of scope and is captured as a follow-up.
- **Auto-fix payloads**: Out of scope. Every rule row declares `auto_fixable` as a flag only, informing Step 5 prioritisation. No `FixSuggestion` data is produced by this spec's code.
- **`.Rnw` / `.Rmd` support**: Out of scope. Rules target `.tex` and `.bib`; the rule `formats` field stays at `None` (all formats), with format narrowing deferred to Step 4.
- **Reuse of existing harness**: Corpus scanning, labelling, and precision reporting reuse the `eval-jss` harness from spec 002 unchanged. This spec does not modify the harness; it only consumes its `report` output as the quality gate.
- **Corpus target for precision-gate enforceability**: Spec 002 starts the corpus at ~10 CRAN vignettes and scopes corpus size as "an operational knob under this spec, not a versioned deliverable." To make the FR-015 N=10-violations-per-rule floor attainable for the majority of rules, this spec grows the corpus to **approximately 50 papers** by the time the catalogue stabilises. The growth is accomplished within spec 002's existing operational flexibility (no spec-002 amendment is required); it is spec 003's task list that schedules the corpus additions alongside each category rollout.
- **Reuse of existing rule API**: The `Rule` / `RuleCheck` / `ParsedDocument` / `Violation` types defined in spec 001 are unchanged here. New rule modules add new `check_*` callables, not new types.
- **Existing smoke rules**: The Step 1 smoke rules (`bib_001_year.py`, `cite_001_emph.py`, `src_001_width.py`) are retrofitted into the catalogue as `JSS-REFS-00N`, `JSS-CITE-00N`, `JSS-CODE-00N` (or similar) rows during the first category pass; they are not treated as a separate tier. Retrofit is a **delete-and-rewrite** migration: when a category that subsumes a smoke rule is introduced, the old one-rule-per-file module and its test are deleted in the same PR that adds the category module, with no compatibility shim and no parallel registration path (see FR-018 / FR-019 / FR-024). The count of new rules in this spec is therefore "approximately 50 − (retrofit count)".
- **Exactness of "50 rules" and "16 categories"**: These are **planning anchors, not contracts.** The final counts are whatever the authorities warrant, filtered through the authority-citation requirement (FR-002) and the precision gate (FR-015). There is no fixed tolerance; catalogue validity is determined by what is in or out of the catalogue, not by how many entries it has. The numerical anchors persist in the spec only as an order-of-magnitude expectation for readers; drift of more than a factor of two from those anchors is an observation worth a clarify pass, but the catalogue's validity is not defined by hitting any count.
- **Authority conflict resolution**: When the prose guide says one thing and `jss.cls` implements another, the class wins and the prose disagreement is recorded in the rule's `notes`. This is the practical consequence of the priority ordering above and needs no separate policy.
