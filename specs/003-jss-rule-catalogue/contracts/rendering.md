# Contract: Catalogue Rendering

**Renderer**: `tools/render_catalogue.py`
**Input**: `specs/003-jss-rule-catalogue/catalogue.yaml`
**Output**: `specs/003-jss-rule-catalogue/catalogue.md`
**Invocation**: `python -m tools.render_catalogue` (from repo root)
**Drift check**: `tests/unit/journals/jss/test_render.py` re-renders and compares byte-for-byte.

## Rendering pipeline

```
catalogue.yaml ──(yaml.safe_load)──► dict ──(sort)──► list[dict] ──(markdown table)──► catalogue.md
```

No templates, no Jinja, no markdown library — string concatenation against the fixed table schema.

## Sort order

Rules are sorted by the tuple `(category_order_index, rule_id_counter_asc)`.

`category_order_index` is the 0-based index of the rule's `category` within the top-level `categories` list of `catalogue.yaml`. This matches the JSSJournal rollout order (see `contracts/rules-module.md §Category rollout order`).

`rule_id_counter_asc` is the integer extracted from the `\d{3}` tail of `rule_id` (`"JSS-CITE-001" → 1`, `"JSS-CITE-042" → 42`).

The sort is stable; ties are impossible given `rule_id` uniqueness.

## Output format

### Header

```markdown
# JSS Rule Catalogue

**Source of truth**: [catalogue.yaml](catalogue.yaml). This markdown is generated by `python -m tools.render_catalogue`; edits here are overwritten.

**Schema version**: 1
**Vendored sources**: `docs/jss-template/jss.cls` dated 2021-05-23
**Rule count**: NN
**Category count**: MM

```

The `NN` and `MM` values are computed from the data. The literal dates come from `source_vendored_at` in the YAML.

### Per-category sections

For each category in `categories` order:

```markdown
## <Category title>

_<category-id>_ — <count> rule(s)

| Rule ID | Severity | Description | Authority | Authority ref | Auto-fixable |
|---|---|---|---|---|---|
| JSS-CITE-001 | warning | \emph used where a citation key is meant; use \cite | style_guide | `#what-are-the-different-cite-citet-citep-commands-about` | ✓ |
| JSS-CITE-002 | ... |
```

**Category title mapping**: the title-case form of `category` with underscores replaced by spaces — `code_style → "Code style"`, `code_width → "Code width"`, `crossrefs → "Cross-references"` (special-cased in the renderer's title-map). Matches `JSSJournal.categories()`'s `RuleCategory.title` values.

**Column details**:

| Column | Source | Rendering |
|---|---|---|
| Rule ID | `rule_id` | verbatim, monospace via backticks in the markdown table: `JSS-CITE-001` |
| Severity | `severity` | verbatim: `error` / `warning` / `info` |
| Description | `description` | verbatim |
| Authority | `authority` | verbatim |
| Authority ref | `authority_ref` | wrapped in backticks: `` `jss.cls:477` `` / `` `#anchor-id` `` |
| Auto-fixable | `auto_fixable` | `✓` if `true`, `—` if `false` |

### Per-rule detail subsections

After the per-category summary table, each rule renders a detail block:

```markdown
### JSS-CITE-001

**Category**: `citations` · **Severity**: `warning` · **Auto-fixable**: no

\emph used where a citation key is meant; use \cite

**Authority**: `style_guide` → `#what-are-the-different-cite-citet-citep-commands-about`

**Inspects**: `tex_files`

<details>
<summary>Example violation</summary>

```latex
Recently \emph{Knuth1984} described algorithms.
```

</details>

<details>
<summary>Example fix</summary>

```latex
Recently \cite{Knuth1984} described algorithms.
```

</details>

**Notes**: Retrofit of the Step 1 smoke rule cite_001_emph.py. …

---
```

- The language tag after the triple-backticks is `latex` when `category` in `{preamble, structure, markup, citations, crossrefs, typography, capitalization, house_style, abbreviations, code_style, code_width, operators, naming}`, and `bibtex` when `category in {references, bibtex}`. The renderer picks this from the category alone; overriding per rule is not supported (Constitution §X — no speculative flexibility).
- The `**Notes**:` paragraph is omitted when `notes` is absent or empty.
- Rules are separated by a horizontal rule (`---`) on its own line.

## Determinism

- YAML keys within each rule are read via `safe_load`; iteration order within a rule follows the emission order above (hard-coded), not the YAML source order.
- No timestamp, no commit hash, no environment-dependent content in the output.
- `python -m tools.render_catalogue` run twice on the same input produces byte-identical output.
- `tests/unit/journals/jss/test_render.py` asserts `render_catalogue(yaml_content) == committed_catalogue_md` using `pathlib.Path.read_text(encoding="utf-8")` on both sides.

## Error handling

The renderer prints to stderr and exits non-zero when:

- `catalogue.yaml` does not exist, is unreadable, or fails `yaml.safe_load`.
- The parsed structure does not match the top-level shape in `catalogue-schema.md §Top-level document` (`version`, `source_vendored_at`, `categories`, `rules` keys).
- Any rule entry fails a catalogue-schema invariant (duplicate `rule_id`, unknown `authority`, unknown `severity`, etc.). The renderer delegates these checks to the same validator used by `test_catalogue.py` (shared helper in `tools/_catalogue_validate.py`).

On success, the renderer writes to `catalogue.md` atomically (via `tempfile.NamedTemporaryFile` + `os.replace`) so a partially-written file can never ship.

## CLI surface

```
usage: python -m tools.render_catalogue [--check]

Options:
  --check   Do not write catalogue.md; exit 0 if the committed markdown
            already matches what would be rendered, 1 if it drifts.
```

`--check` is what `test_render.py` uses internally (and what a CI job would invoke as a sanity check outside pytest). Default (no flag) writes the file.

## Why no template engine

Constitution §X — small surface. A template engine's dependency tree, version churn, and API surface are larger than this renderer. The table shape is fixed; the per-rule detail block is fixed; neither will ever need user-configurable layout.

## Why markdown and not HTML

The catalogue's consumers are GitHub PR reviewers, who render markdown natively. HTML would require either a hosted Pages site or an offline preview step, neither of which is worth the complexity. Raw markdown is also the most diff-friendly format: a PR that adds a rule shows exactly one insertion block in the table and one detail block.
